# ADR-001. Размещение оркестратора

## Общая информация

1. Родительский артефакт - Архитектура платежной системы
2. Автор ADR - Кудрин О.А.
3. Статус - В проработке
4. Согласующие - Начальство

## Контекст

Проработан бизнес-процесс обработки платежа на маркетплейсе в виде Saga-оркестрации,
в варианте Parallel Saga (асинхронная коммуникация, итоговая согласованность,
координаяция с помощью оркестрации). Теперь нужно решить где разместить функцию
оркестрации, внутри одного из доменных сервисов или в отдельном сервисе-оркестраторе.

Ключевые моменты:

- сложность бизнес-процесса;
- компенсационные транзакции;
- state machine для управления Saga;
- интеграция с внешними API;
- мониторинг и трассировка.

## Рассмотренные варианты

### Вариант 1. Отдельный оркестратор

Создаем выделенный микросервис, который управляет шагами Saga,
отслеживает состояние, рассылает команды.

### Вариант 2. Встроенная оркестарция

Один из сервисов доменной области берет на себя роль координатора,
вызывая других участников и обрабатывая их ответы.

## Сравнение альтернатив

Отдельный оркестратор:

- Преимущества
  - централизованное управление
  - лучшие наблюдаемость и аудит
  - лучшая гибкость при изменении безнес-процессов
- Недостатки
  - дополнительный сервис
  - дополнительная инфраструктура
  - дополнительная поддержка

Встроенная оркестрация:

- Преимущества
  - проще в реализации
  - быстрее по времени отклика
- Недостатки
  - приводит к разрастанию сервисов (бог-сервис)
  - усложнение поддержики при увеличении числа сценариев

Взвесим достоинства обоих вариантов:

| Отдельный оркестратор | Встроенная оркестрация |
| --------------------- | ---------------------- |
| инкапсуляция логики   | простота системы       |
| развитие системы      | производительность     |
| переиспользование     |                        |
| наблюдаемость и аудит |                        |

## Принятое решение

Решено выбрать вариант 1, отдельный оркестратор.

Причины:

- В процессе участвуют три и более сервисов.
- Есть множество разветвленных веток.
- Требуются аудит, визуализация прогресса, операторские действия, ручные компенсирующие шаги.
- Возможно потребуется переиспользование наработок процесса
- Возможны изменения сценариев, версионирование, эксперименты.
